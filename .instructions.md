# JMS Tax Project Standards

This project is an Electron desktop application with React frontend and Supabase backend for tax consultancy management.

## Architecture Rules

### 1. Component Organization

**ALWAYS use existing components from the components folder. Never create new components in pages.**

Available component categories:

- `components/ui/` — Button, TextField, SelectField, CheckboxField, Card, Chip, IconButton, DropdownMenu, DropZone, Modal, ConfirmDialog
- `components/common/` — EmptyState, ServiceCard, LoadingSpinner, StatCard, FloatingActionBar
- `components/layout/` — Sidebar, Header, AppLayout, TabBar
- `components/table/` — DataTable, Pagination

**Rules:**

- ALWAYS check existing components before creating new ones
- New components MUST be created in the appropriate subfolder under `src/renderer/components/`
- Components should be reusable and generic
- Use PascalCase for component file names (e.g., `UserCard.tsx`)

### 2. Hooks Pattern

**NO direct database calls in pages or components. ALWAYS use hooks.**

Available hooks:

- `useNetworkStatus.ts` — Network connectivity and online/offline status
- `useTaxRecords.ts` — Tax records CRUD, filtering, bulk operations
- `usePortalPages.ts` — Portal management CRUD
- `useTabNavigate.ts` — Tab-based navigation state
- `useUpdater.ts` — Electron auto-updater integration

**Rules:**

- ALL data fetching MUST go through hooks in `src/renderer/hooks/`
- Hooks MUST use React Query (@tanstack/react-query) for server state
- Use camelCase for hook file names (e.g., `useUserData.ts`, `useClients.ts`)
- Hooks should encapsulate:
  - Supabase queries
  - Loading states
  - Error handling
  - Data transformations

**Hook Template:**

```typescript
import { useQuery, useMutation } from '@tanstack/react-query';
import { supabase } from '@/lib/supabase';

export function useExampleData() {
  const { data, isLoading, error } = useQuery({
    queryKey: ['example-key'],
    queryFn: async () => {
      const { data, error } = await supabase.from('table').select();
      if (error) throw error;
      return data;
    },
  });

  return { data, isLoading, error };
}
```

### 3. Pages Structure

**MUST use folder-based routing, NOT file-based routing.**

Current problematic structure (DO NOT FOLLOW):

```
pages/
  dashboard/Dashboard.tsx ❌
  tax-records/TaxRecords.tsx ❌
```

Correct structure (ALWAYS USE):

```
pages/
  dashboard/
    index.tsx ✅
    components/
      StatsGrid.tsx
      RecentActivity.tsx
  tax-records/
    index.tsx ✅
    new/
      index.tsx
    [id]/
      index.tsx
      edit/
        index.tsx
    components/
      RecordTable.tsx
```

**Rules:**

- Each page MUST have its own folder with `index.tsx`
- Sub-pages go in nested folders with their own `index.tsx`
- Use kebab-case for folder names (e.g., `tax-records/`, `client-management/`)
- Page-specific components go in a `components/` subfolder within the page folder
- Main page component should be named simply (not `DashboardPage.tsx`, just export from `index.tsx`)

**Page Template:**

```typescript
// pages/dashboard/index.tsx
export default function Dashboard() {
  return <div>Dashboard content</div>;
}
```

### 4. Naming Conventions

**Strict naming rules:**

- **Folders:** kebab-case (`tax-records`, `client-management`, `user-settings`)
- **Hooks:** camelCase (`useTaxRecords.ts`, `useUserProfile.ts`, `useClientData.ts`)
- **Variables:** camelCase (`taxRecord`, `clientList`, `isLoading`)
- **Components:** PascalCase (`UserCard.tsx`, `StatCard.tsx`, `DataTable.tsx`)
- **Constants:** UPPER_SNAKE_CASE (`MAX_FILE_SIZE`, `API_ENDPOINT`)
- **Types/Interfaces:** PascalCase (`TaxRecord`, `ClientData`, `UserProfile`)

### 5. UI Library Standards

**ONLY use approved UI libraries:**

- **Icons:** ONLY use `@heroicons/react` (no other icon libraries)
  - Use `/24/outline` or `/24/solid` for standard sizes
  - Use `/20/solid` for compact/inline icons (table actions, filter bars)
- **UI Primitives:** `@headlessui/react` is used internally by `DropdownMenu` and `Modal` only — do not use HeadlessUI directly in pages or new components
- **Styling:** TailwindCSS only (no CSS modules, no styled-components)
- **Notifications:** Use `sonner` for toast notifications

**Rules:**

- **Never use a native `<select>` element.** Always use `SelectField` from `@components/ui/SelectField`
- **Never use a raw `<button>` for actions.** Use `Button`, `IconButton`, or `DropdownMenu`
- All components are default exports — use `import Button from '@components/ui/Button'`, not `import { Button }`
- Use existing UI components from `components/ui/` before creating new ones
- All styling MUST use Tailwind utility classes
- Responsive design: mobile-first approach

**Example:**

```typescript
import { PlusIcon } from '@heroicons/react/20/solid';
import Button from '@components/ui/Button';
import SelectField from '@components/ui/SelectField';
import { toast } from 'sonner';

export default function Example() {
  const [status, setStatus] = useState('active');

  return (
    <>
      <SelectField
        id="status"
        label="Status"
        value={status}
        onChange={setStatus}
        options={[
          { value: 'active', label: 'Active' },
          { value: 'inactive', label: 'Inactive' },
        ]}
      />
      <Button icon={PlusIcon} onClick={() => toast.success('Saved!')}>
        Add Record
      </Button>
    </>
  );
}
```

### 6. Code Quality Standards

**Maintain high code dignity:**

- Write clean, readable, self-documenting code
- Use TypeScript strictly (no `any` types without justification)
- Keep functions small and focused (single responsibility)
- Extract complex logic into utility functions in `src/renderer/lib/`
- Add comments only for complex business logic, not obvious code
- Use meaningful variable and function names
- Handle errors gracefully with user-friendly messages
- Always validate user input
- Implement loading and error states

**Error Handling Template:**

```typescript
import { toast } from 'sonner';

try {
  await saveTaxRecord(data);
  toast.success('Tax record saved successfully');
} catch (error) {
  console.error('Failed to save tax record:', error);
  toast.error('Failed to save tax record. Please try again.');
}
```

### 7. Data Layer — camelCase Convention

**All Supabase data is camelCase on the frontend. Never use snake_case field names in components, hooks, or services.**

**Utilities:** `src/renderer/lib/caseTransform.ts`

- `toCamelCase(obj)` — convert Supabase snake_case response → camelCase (use in `mapRow`)
- `toSnakeCase(obj)` — convert camelCase payload → snake_case (use before `.insert()` / `.update()`)

**Reading from Supabase:**

```typescript
function mapRow(row: Record<string, unknown>): TaxRecord {
  const r = toCamelCase(row);
  return { id: r.id as number, referenceNumber: r.referenceNumber as string, ... };
}
```

**Writing to Supabase:**

```typescript
// Spread camelCase payload — toSnakeCase handles the key conversion
supabase.from('tax_records').insert(toSnakeCase({ ...payload, userId }));
supabase
  .from('tax_records')
  .update(toSnakeCase({ ...payload, updatedAt: new Date().toISOString() }));
```

**Auth / user profile:**

- `AuthContext` exposes `userInfo: UserInfo | null` from `useAuth()` — use this for all auth and profile data, never access `user` or `user.user_metadata` directly
- `UserInfo` fields: `email`, `createdAt`, `provider`, `fullName`, `companyName`, `address`, `phoneNumber`, `description`
- Writes use `toSnakeCase()` inside `AuthContext.updateProfile()`

**Uniqueness validation:**

- `email`, `cnic`, `reference_number` are unique **per user** (composite DB constraints with `user_id`), not globally
- `taxRecordApi.validateUniqueness()` runs all checks in parallel and returns all errors at once

### 8. Enums-First Policy

**Use centralized enums/constants for all table names, edge function names, module names, audit actions/events, pagination sizes, intervals, and query keys. Avoid hardcoded string literals.**

**Source:** `src/renderer/lib/enums/index.ts`

```typescript
import {
  TABLES,
  EDGE_FUNCTIONS,
  MODULES,
  AUDIT_ACTIONS,
  AUDIT_EVENTS,
  PAGE_KEYS,
  PAGE_SIZE,
  INTERVALS,
} from '@lib/enums';
```

**Rules:**

- Use `TABLES.*` in every Supabase `.from()` call
- Use `EDGE_FUNCTIONS.*` in every `supabase.functions.invoke()` call
- Use `MODULES.*` when writing or reading audit logs
- Use `AUDIT_ACTIONS.*` for action values (`create`, `update`, `delete`, bulk/export actions)
- Use `AUDIT_EVENTS.*` for snapshot event keys
- Use `PAGE_KEYS.*` for React Query keys and persisted page keys
- Use `PAGE_SIZE.*` and `INTERVALS.*` instead of numeric literals where applicable

### 9. Audit Trail Standard (All Modules)

**Every mutating flow must write audit trail entries.**

Modules currently covered in enums:

- `MODULES.TAX_RECORD`
- `MODULES.PORTAL_PAGE`
- `MODULES.AUTH`
- `MODULES.TEAM_MANAGEMENT`
- `MODULES.USER_PERMISSIONS`

**When to log:**

- Create: log with `AUDIT_ACTIONS.CREATE`
- Update: compute field diff and log with `AUDIT_ACTIONS.UPDATE`
- Delete: include pre-delete snapshot with `AUDIT_ACTIONS.DELETE`
- Bulk actions: use matching `AUDIT_ACTIONS.BULK_*`
- Exports: use `AUDIT_ACTIONS.EXPORT_PDF` / `AUDIT_ACTIONS.EXPORT_CSV` with `AUDIT_EVENTS.*` snapshot event

**Required payload fields:**

- `module`: from `MODULES.*`
- `recordId`: string id (or `null` for cross-record actions)
- `action`: from `AUDIT_ACTIONS.*`
- `changedByName`
- `changes`: diff object for updates
- `snapshot`: record state or event metadata (for deletes/exports/bulk operations)

**Implementation rules:**

- Call `auditLogApi.log(...)` in hooks/services that execute mutations, never directly from UI components
- Use `diffRecord(before, after)` for update actions; skip log only if diff is `null`
- Audit log failures must not block the primary operation (existing `auditLogApi.log` behavior)
- Keep audit log module values and action values enum-backed, not free-form strings

### 10. Supabase Integration

**All database operations through hooks:**

- Use `@supabase/supabase-js` client from `src/renderer/lib/supabase`
- Leverage React Query for caching and synchronization
- Use real-time subscriptions when needed
- Implement optimistic updates for better UX

**Example:**

```typescript
const { mutate: createRecord } = useMutation({
  mutationFn: async (data: TaxRecord) => {
    const { data: result, error } = await supabase
      .from('tax_records')
      .insert(data)
      .select()
      .single();
    if (error) throw error;
    return result;
  },
  onSuccess: () => {
    queryClient.invalidateQueries({ queryKey: ['tax-records'] });

  },
});
```

## File Structure Reference

```
src/
  renderer/
    components/          # Reusable components only
      ui/               # Basic UI elements
      common/           # Shared business components
      layout/           # Layout components
      table/            # Table-related components
      modals/           # Modal dialogs
    hooks/              # Custom React hooks (all data fetching)
    pages/              # Page components (folder-based)
      [page-name]/
        index.tsx       # Main page component
        components/     # Page-specific components
        [sub-page]/
          index.tsx
    lib/                # Utility functions
    services/           # API services (if needed)
    contexts/           # React contexts
    types/              # TypeScript types
  main/                 # Electron main process
  shared/               # Shared code between main and renderer
```

## Migration Plan

When refactoring existing code to match these standards:

1. Move page components to folder structure with `index.tsx`
2. Extract direct Supabase calls to hooks
3. Identify reusable components and move to `components/`
4. Rename files to match naming conventions
5. Update imports throughout the codebase
6. Test thoroughly after refactoring
